# OSCAL — Open Security Controls Assessment Language

ReguNav emits framework dictionaries in **OSCAL 1.1.2** form so they can be
consumed by any OSCAL-aware tool — most notably US federal / FedRAMP
pipelines — without bespoke parsing.

OSCAL is a NIST standard. The authoritative website is
[https://pages.nist.gov/OSCAL/](https://pages.nist.gov/OSCAL/), with
metaschema, JSON Schema, examples and tooling.

---

## What we emit

For every populated framework in `@regunav/frameworks` (21 today —
ISO/EU/US/UK/global authorities) the emitter writes two OSCAL files:

| Artefact | What it represents | Built from |
| --- | --- | --- |
| **Catalog** | The framework's clauses, expressed as OSCAL `control`s grouped under a single `group`. Each clause carries a `statement` part (the description) and, where any control satisfies it, a `guidance` part listing the satisfying ReguNav controls. | `Framework.clauses[]` (with `Framework.controls[].clauseRefs[]` indexed to produce guidance) |
| **Component-Definition** | The framework's controls, expressed as OSCAL `component`s of `type: "policy"`. Each component carries one `control-implementation` whose `implemented-requirements[]` map every `clauseRef` the control satisfies back to the catalog's control ids. | `Framework.controls[]` |

Both files live under `oscal-emit/` (gitignored — regenerate with the
script). The pair round-trips: the component-definition's
`control-implementation.source` points at the sibling catalog file, and
every `implemented-requirements[*].control-id` resolves to a control in
that catalog.

Output paths:

```
oscal-emit/
  INDEX.md                                  ← lists every file + size + SHA-256
  <framework-code>.catalog.json             ← OSCAL Catalog
  <framework-code>.component-definition.json ← OSCAL Component-Definition
  ...                                       ← 2 files per populated framework
```

The canonical TypeScript emitter lives at
`packages/ontology/src/oscal-emit.ts`. The bulk emitter script at
`scripts/oscal/emit-all-frameworks.mjs` mirrors the same logic so the
artefacts can be regenerated without TS tooling.

## What we do NOT emit

The following OSCAL layers exist in the spec but are intentionally **not**
generated by this emitter because they require tenant-specific data:

| Artefact | Why it's not framework-static |
| --- | --- |
| **Profile** | A profile selects/tailors controls. The selection set depends on which obligations apply to a given organisation — that's the job of `ApplicabilityRule` in the C2A ontology, run against tenant context. |
| **Assessment Plan** | Describes the methods (`EXAMINE`/`INTERVIEW`/`TEST`) and sampling strategy for testing controls. Needs the tenant's chosen `ControlTest` configuration. |
| **Assessment Results** | Captures observations + findings from running an assessment. Needs the tenant's collected `EvidenceObject`s and test outcomes. |
| **POA&M** (Plan of Action & Milestones) | Tracks open findings and remediation. Needs the tenant's open exceptions and `Mitigation` records. |

For the C2A ↔ OSCAL mapping that DOES use tenant data, see the runtime
emitters in `packages/ontology/src/oscal.ts` (`exportProfile`,
`exportComponentDefinition`, `exportPoam`,
`evidenceToOscalObservation`).

## How a consumer uses our emit

A typical OSCAL consumer (e.g. a GRC tool, a FedRAMP package builder,
or a custom audit pipeline) can:

1. **Ingest the catalog** (`<code>.catalog.json`) to learn the framework's
   clause taxonomy. Each clause becomes a `control` with a stable
   deterministic UUID. Custom metadata is carried in
   `props[*]{name,value,ns: "https://regunav.com/oscal-ns/v0.1"}` —
   notably `framework-code`, `clause-ref`, `mandatory` and
   `jurisdiction`.

2. **Build a profile** that selects only the catalog controls relevant
   to their system (e.g. via `include-controls.with-ids`).

3. **Ingest the component-definition** (`<code>.component-definition.json`)
   to discover the ReguNav implementation guidance. Each component's
   `control-implementations[*].implemented-requirements[*].control-id`
   references a control in the catalog, so the consumer can link a
   selected control (from step 2) to one or more recommended
   implementations.

4. **Author their own component-definition** for their actual
   implementation, copying or referencing the emitted requirements as
   a starting point.

5. **Generate an Assessment Plan / Results / POA&M** using their own
   tenant data — these are intentionally not in our emit (see above).

## Identifiers

OSCAL requires UUIDs on most top-level objects. To make the emit
**deterministic** (re-runs produce byte-identical output except for
the `metadata.last-modified` timestamp), the emitter derives every UUID
from a SHA-256 of stable inputs:

| UUID | Derived from |
| --- | --- |
| Catalog UUID | `("catalog", framework.code, framework.version)` |
| Catalog resource UUID | `("resource", framework.code, framework.referenceUrl)` |
| Component-Definition UUID | `("component-definition", framework.code, framework.version)` |
| Component UUID | `("component", framework.code, control.controlRef)` |
| Control-Implementation UUID | `("control-implementation", framework.code, control.controlRef, catalogHref)` |
| Implemented-Requirement UUID | `("implemented-requirement", framework.code, control.controlRef, clauseRef)` |

The output is formatted as a v5-style UUID (8-4-4-4-12 with version
nibble 5 and RFC 4122 variant nibble). OSCAL validators that check the
UUID layout will accept these; consumers that require strict RFC 4122
v5 (SHA-1 over a namespace) can re-key on ingestion using the same
inputs.

## Provenance and disclaimer

- The framework clause prose and control descriptions are derived from
  the published source authority (cited via `back-matter.resources` and
  per-group `links`). They are structural references to the regulation,
  not legal advice. See `packages/ontology/LEGAL_DISCLAIMER.md`.
- Generated examples in `oscal-emit/` are **sample/synthetic** output of
  the emitter. The substantive content (clauses, controls) is
  authoritative; the OSCAL wrapping is mechanical.
- No ™ / ® / © glyphs are emitted into OSCAL output. NIST OSCAL examples
  do not use them and they make string equality / diff brittle across
  encodings.

## Regenerating the emit

```bash
pnpm install
pnpm --filter @regunav/frameworks build   # ensure dist/index.js is current
node scripts/oscal/emit-all-frameworks.mjs
```

The script prints one line per framework plus a final summary, and
writes `oscal-emit/INDEX.md` with the full file inventory.
